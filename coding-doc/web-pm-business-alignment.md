# Web PM Business Alignment

最後更新：2026-03-05  
狀態：啟用中（作為功能需求提出與開發對齊基線）

## 1) 文件用途

本文件從產品經理（PM）角度定義業務理解與功能對齊規則，避免功能開發偏離客戶真實業務需求。  
當提出新功能或變更需求時，開發必須先依本文件完成業務映射，再進入設計與實作。

## 2) 口徑與資料來源

1. 業務背景與術語來源：`coding-doc/web-business-reference.md`（由 `reference-doc/` 維護）。
2. 功能現況來源：`coding-doc/web-functions.md` + `web/` 現有程式實作（唯一功能基線）。
3. 舊版 `doc/docs/` 內項目背景、功能排期、模組範圍、故事優先級不作為當前功能真實依據。

## 3) 客戶業務背景（摘要）

核心業務痛點：

- 地界與實際占用範圍不一致，人工判斷成本高且容易誤判。
- 地塊、住戶、狀態、任務資訊分散，跨角色協作效率低。
- 現場作業缺乏地圖化、可追蹤、可回查的統一工作介面。
- 新增時程語意（`part of site` / `section of works`）後，日期推導規則更複雜，若無結構化建模容易口徑漂移。

本產品（現階段 web）要提供的業務價值：

- 以地圖為核心整合地塊作業、清單管理、進度觀測與匯出能力。
- 讓管理與現場角色可在同一資料語意下溝通（Work Lot / Site Boundary）。
- 讓風險與進度狀態可被看見、可被追蹤、可被導出。
- 在 Demo 階段，提醒以網站內可視化呈現為主，降低跨工具溝通成本。

## 4) 業務需求到現有功能的對應

| 業務需求 | 現有 web 功能（基線） | 開發新增/變更時必查 |
| --- | --- | --- |
| 快速定位與圈選目標範圍 | `/map` scope、搜尋、圖層控制 | 是否維持 map 主要工作流可達、快捷鍵不衝突 |
| 現場/管理對同一地塊狀態認知一致 | Work Lot / Site Boundary 清單 + map 抽屜詳情 | 狀態語意、欄位命名、顯示位置是否一致 |
| 追蹤進度與風險 | `/dashboard` KPI 與趨勢、boundary 狀態彙總 | 風險判定邏輯是否可解釋且與現有指標一致 |
| `access date` 到期提醒（Web in-app） | 地圖與清單已有逾期視覺語意（基於 `dueDate` / `plannedHandoverDate`） | 是否新增 `accessDate` 與專屬到期規則，並明確呈現到期原因 |
| Topbar 鈴鐺通知中心（Web in-app） | 已支援通知抽屜（`All / Alert / Task / System`）、未讀管理、一鍵跳轉 Map/清單 | 通知是否覆蓋核心風險、分類是否可理解、是否保留 email 擴展位 |
| `part of site` / `section of works` 時程管理 | 現有可維護日期欄位（assess/handover/completion/float） | 是否補齊結構欄位與映射規則，避免把不同時程語意混在單一欄位 |
| `site/work lot/part/section` C1/C2 分層作業（不對照） | 現有已改為 `Contract Workspace` 單一切換（C1/C2） | 是否維持 map/清單/scope/匯出一致套用同一 `activeContract` 口徑 |
| `part of site` 圖面修補與上圖 | 現有可透過 DXF 轉 GeoJSON 腳本與地圖圖層顯示達成 | 是否需要 CAD 級編輯（刪線段/刪 block/補線）；若不做，需定義前處理工具與責任人 |
| 需要可交付資料給外部單位 | JSON 匯入匯出、Excel/PDF 報表 | 匯出欄位是否滿足業務口徑、格式是否可用 |
| 多角色操作邊界清晰 | 前端角色切換、路由守衛、地圖編輯限制 | 是否新增越權操作、deny 行為是否明確 |

## 5) PM 提需求最小欄位（開發前置輸入）

每個需求至少要包含：

1. 業務目標：要解決哪個痛點，影響哪個角色。
2. 業務場景：在什麼作業節點使用（例：地圖圈選、清單編修、報表輸出）。
3. 成功判準：如何判斷此功能真的解決問題（可量化優先）。
4. 資料影響：是否新增欄位、狀態、匯入匯出格式。
5. 權限影響：哪些角色允許/禁止，阻擋時 UI 行為。
6. 時程規則來源：是否來自合約條款、客戶表格或會議決議，起算日與計算公式為何。
7. 通知渠道策略：Web in-app、Email 或雙軌；哪個階段啟用。
8. 圖面品質與前處理：是否需先在 CAD/腳本完成刪線、去 block、補線閉合；哪些屬性（如 `part name`/`access date`）由人工補錄。
9. 階段分層口徑：若有 C1/C2，需明確欄位命名、同一 ID 跨階段是否共用、以及圖層/匯出顯示規則。

## 6) 開發對齊流程（強制）

1. 先讀本文件 + `coding-doc/web-functions.md`，確認需求對應的業務價值鏈。
2. 盤點受影響模組：路由/頁面、資料契約、權限、匯出流程。
3. 先寫「業務對齊說明」再開發：
   - 解決哪個業務痛點
   - 影響哪個角色與作業節點
   - 與現有功能差異是新增、替換或補強
4. 完成實作後，同步更新對應 `coding-doc` 文件，確保文件與功能一致。
5. 驗證必須同時涵蓋：
   - 功能可用性
   - 業務語意一致性
   - 權限與資料一致性

## 7) 業務一致性驗收清單

1. 新功能能清楚對應至少 1 個業務痛點，且不是純 UI 裝飾。
2. 需求描述中的角色、名詞、狀態與系統現有語意一致。
3. 若有新欄位/狀態，`web-data-contract.md` 已同步更新。
4. 若有權限變更，`web-role-permission-spec.md` 已同步更新。
5. 若影響使用者可見流程，`web-functions.md` 已同步更新。
6. 若涉及時程推導，驗收結果需明確列出「起算日、計算規則、覆核方式」。
7. 若涉及提醒，需驗證 Web 呈現位置、觸發條件與關閉條件一致。
8. 需求完成回報包含「業務價值是否成立」的結論。
9. 若涉及 C1/C2，需驗證 workspace 切換後只顯示單一 contract，且同一業務 ID 在不同 phase 仍可分開保存不互相覆蓋。
